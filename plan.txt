1. HIGH-LEVEL ARCHITECTURE
================================================================================

Pipeline (left to right):

  BASELINE → FINE-TUNING → OPTIMIZATION → INFERENCE → BENCHMARKING

1) BASELINE
   - Load a small instruction-following model from Hugging Face (Transformers).
   - Run a fixed prompt set on CPU and GPU (optional) to establish latency,
     peak memory, and qualitative baselines before any training.
   - Freeze all weights; no adaptation yet.

2) FINE-TUNING (SFT, LoRA, QLoRA)
   - Prepare a compact, high-signal dataset (e.g., instruction–response pairs).
   - SFT: full or selective fine-tuning of trainable layers (keep scope small).
   - LoRA: inject low-rank adapters via PEFT; base weights frozen.
   - QLoRA: 4-bit (or 8-bit) base model + LoRA adapters; train adapters only.
   - Each run writes: config snapshot, metrics, checkpoints path, logs.

3) OPTIMIZATION (post-training compression / serving prep)
   - Quantize trained artifacts for inference (8-bit, 4-bit) where supported.
   - Compare memory footprint and throughput vs unquantized adapters.
   - Document tradeoffs: accuracy drift vs speed and VRAM.

4) INFERENCE
   - FastAPI service exposing POST /generate with JSON body:
     model_kind (base | sft | lora | qlora | quantized variants),
     prompt, generation parameters (max_new_tokens, temperature, top_p).
   - Load models lazily or via a small registry mapping names to loaders
     (implementation detail left for coding phase; this plan stays conceptual).

5) BENCHMARKING
   - Same fixed prompts and decoding settings across all variants.
   - Record: wall-clock latency (p50/p95 if repeated), peak RAM/VRAM,
     tokens per second (optional), and lightweight quality checks
     (exact match on tiny held-out set, or LLM-as-judge only if you add it later).
   - Emit a structured table (CSV or JSON) plus human-readable summary.

Data and control flow (conceptual):

  Config files → Training scripts → Checkpoints + metrics
  Checkpoints + config → API loaders → /generate
  Benchmark driver → invokes API or local generate → results table

Why this order: baseline isolates hardware effects; fine-tuning proves adaptation;
optimization proves deployment thinking; inference proves systems integration;
benchmarking proves measurement discipline.

3. STEP-BY-STEP EXECUTION PLAN (PHASES)
================================================================================

--------------------------------------------------------------------------------
PHASE 1: SETUP + BASELINE
--------------------------------------------------------------------------------
What to implement:
- Python environment, GPU driver sanity check, pinned dependencies.
- Config schema for model name, dtype, device, and paths.
- Baseline script: load model, run N fixed prompts, log time and memory.
- Minimal logging setup (structured log lines or JSON lines).

Expected output:
- Baseline metrics file (e.g., experiments/baseline_<date>/metrics.json).
- Confirmed working path from Hugging Face model id to local generate.

Key learning:
- End-to-end reproducibility starts with environment and pinned configs.
- Baseline isolates inference cost before any training variables.

--------------------------------------------------------------------------------
PHASE 2: SUPERVISED FINE-TUNING (SFT)
--------------------------------------------------------------------------------
What to implement:
- Instruction dataset in a simple format (JSONL: instruction, response).
- Training script using Hugging Face Trainer (or similar) with args from config.
- Train/eval split; short training run (demo-scale, not leaderboard-scale).
- Save checkpoint and training curves (loss).

Expected output:
- Adapter-free fine-tuned weights (or partial fine-tune, per config).
- Training log with steps, loss, and wall time.

Key learning:
- SFT is the reference for “full update” behavior vs parameter-efficient methods.
- Data quality and length dominate small-model results.

--------------------------------------------------------------------------------
PHASE 3: LORA (PEFT)
--------------------------------------------------------------------------------
What to implement:
- Same dataset and comparable hyperparameters where fair.
- PEFT LoRA config: rank, alpha, target modules documented in config file.
- Train only adapters; merge or save adapter weights separately.

Expected output:
- Adapter checkpoint directory (small on disk vs full model).
- Metrics comparable to SFT phase for fair benchmarking later.

Key learning:
- LoRA reduces trainable parameters and often matches SFT for narrow tasks.
- Target module choice matters for transformer blocks.

--------------------------------------------------------------------------------
PHASE 4: QLORA
--------------------------------------------------------------------------------
What to implement:
- Load base model in 4-bit (BitsAndBytes) with PEFT LoRA on top.
- Training with gradient checkpointing if needed (config flag).
- Save adapters; record training peak VRAM vs standard LoRA.

Expected output:
- QLoRA adapter bundle + training metrics emphasizing memory savings.

Key learning:
- Quantization for training is different from quantization for inference;
  QLoRA demonstrates training under memory pressure.

--------------------------------------------------------------------------------
PHASE 5: GPU EXPERIMENTS
--------------------------------------------------------------------------------
What to implement:
- Controlled sweeps (small grid): batch size, gradient accumulation steps,
  mixed precision off vs fp16 vs bf16 (hardware permitting).
- Instrumentation: step time, throughput, peak allocated memory.

Expected output:
- A small results matrix (per experiment id) appended to registry.
- Short notes on OOM failures and which config recovered via accumulation.

Key learning:
- Effective batch size = batch_size × accumulation × num_gpus (conceptually).
- Mixed precision trades numerical range for speed; bf16 vs fp16 depends on GPU.

--------------------------------------------------------------------------------
PHASE 6: QUANTIZATION (INFERENCE)
--------------------------------------------------------------------------------
What to implement:
- Post-training quantization paths: 8-bit and 4-bit inference models
  (where the stack supports it cleanly).
- Side-by-side load of base, fine-tuned, and quantized variants for compare.

Expected output:
- Quantized artifact paths and a short report of load memory and latency.

Key learning:
- Quantization is a serving concern; measure quality impact on your task.

--------------------------------------------------------------------------------
PHASE 7: API (FASTAPI)
--------------------------------------------------------------------------------
What to implement:
- FastAPI app with POST /generate.
- Request validation (prompt length limits, parameter bounds).
- Model routing: base, sft, lora, qlora, quantized (names aligned with registry).
- Health endpoint for orchestration (e.g., GET /health).

Expected output:
- Running server documented with example curl or HTTP client calls.
- Consistent JSON response shape (text, model_used, timing metadata optional).

Key learning:
- Production APIs separate transport from model logic and enforce limits.

--------------------------------------------------------------------------------
PHASE 8: BENCHMARKING
--------------------------------------------------------------------------------
What to implement:
- Benchmark driver hitting /generate or calling shared inference module.
- Repeated requests for latency percentiles; memory sampling where possible.
- Quality checks: small golden set or simple metric (per project choice).

Expected output:
- Structured results table (see section 6) exported as CSV/JSON.
- Narrative summary: best latency, best quality, best memory tradeoff.

Key learning:
- Benchmarks are only fair with fixed prompts, temperature, and max tokens.
- p95 latency often matters more than mean for interactive UX.

5. EXPERIMENT DESIGN
================================================================================

Naming: Each experiment gets an ID recorded in registry.txt (see registry file).

E1 — CPU vs GPU inference (baseline)
- Independent variables: device = cpu | cuda
- Fixed: model, prompts, max_new_tokens, dtype policy
- Measured: latency, peak memory, tokens/sec
- Hypothesis: GPU lowers latency for autoregressive decode at scale; CPU viable for tiny models only.

E2 — Batch size vs memory (training)
- Independent variables: per_device_train_batch_size
- Fixed: model, seq length, accumulation=1 initially
- Measured: peak VRAM, step time, OOM boundary
- Hypothesis: Larger micro-batches increase memory linearly until OOM.

E3 — Gradient accumulation (effective batch)
- Independent variables: gradient_accumulation_steps
- Fixed: per_device_train_batch_size at a safe value
- Measured: step wall time, training stability (loss curve)
- Hypothesis: Accumulation simulates larger batch with similar updates, slower steps.

E4 — FP32 vs FP16 vs BF16 (training or inference)
- Independent variables: fp16, bf16 flags (mutually exclusive where required)
- Fixed: model, batch, accumulation
- Measured: throughput, peak memory, any NaN/instability flags
- Hypothesis: Mixed precision improves throughput; bf16 often more stable on supported GPUs.

E5 — LoRA rank sweep (optional small grid)
- Independent variables: r in {8, 16, 32}
- Fixed: data, steps, alpha scaled consistently if desired
- Measured: val loss proxy, adapter size, inference latency overhead

E6 — 8-bit vs 4-bit inference quantization
- Independent variables: quantization mode for inference load
- Fixed: prompts, decoding params
- Measured: load time, VRAM, latency, qualitative output comparison

E7 — Model variant quality spot-check
- Independent variables: base | sft | lora | qlora | quantized
- Fixed: prompts and decoding
- Measured: simple automatic score or human rubric on a tiny set

Always document: hardware model, CUDA version, library versions, and random seed.
